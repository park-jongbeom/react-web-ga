## 🎨 3. UI/UX 우선순위

### 3.1 매칭 결과 시각화 (최우선)

> **"맞춤형 AI 프로그램이 시각적으로 명확하게 보여져야 차별성이 있다"** - JOO LIM

**상세 화면 구성 및 Gap 분석**: [matching-result-design-spec.md](./matching-result-design-spec.md) 참조 (레퍼런스 이미지 기반 6개 섹션, 구현 로드맵).

#### 필수 표시 요소
- **매칭 퍼센트**: 전체 매칭률 표시
- **카테고리별 매칭 결과**:
  - 예산 적합도
  - 전공 적합도
  - 성향 적합도
  - 지역 선호도
  - 학업 기간
  - 진로 연계성
- **매칭 이유**: 각 카테고리별 설명

#### 비주얼 표현 방식
> **"로드맵 그래프는 아니더라도 예산, 전공, 성향 등 정해진 카테고리에 맞춰 매칭 퍼센트와 이유를 보여주는 방식이 비주얼적으로 중요하다"** - Tim App

- Radar Chart 또는 Bar Chart 활용
- 직관적인 색상 코딩 (높음: 녹색, 중간: 노란색, 낮음: 회색)
- 각 지표별 상세 설명 툴팁

### 3.2 추천 학교 리스트

#### 표시 내용
- Top 5~10 추천 학교
- 학교명, 프로그램 유형, 매칭률

## 💻 5. 기술적 결정사항

### 5.1 AI 기능 구현 전략

#### RAG vs 프롬프트 엔지니어링

**현재 상황**
- Patrick Park이 Spring AI 기반 RAG 구현 완료
- 그러나 RAG는 정확한 데이터가 필요

**회의 결정: 프롬프트 엔지니어링 우선**
> **"현재 GPU 자원과 데이터 구축이 부족한 상황이므로, 비용이 가장 적게 드는 프롬프트 엔지니어링을 활용하여 AI 에이전트에게 일을 시키는 방법을 추천"** - Tim App

#### 단계적 접근
1. **1단계 (프로토타입)**: 프롬프트 엔지니어링으로 Gemini API 활용
2. **2단계 (데이터 확보 후)**: RAG 데이터셋 구축
3. **3단계 (장기)**: 자체 모델 파인튜닝

### 5.2 Gemini API 활용 방안

#### 챗봇 인터페이스 구현
> **"현재로서는 데이터베이스가 구축될 때까지는 일반 Gemini API를 활용하여 챗봇 인터페이스를 보여주는 것이 최선"** - Tim App

**구현 범위**
- 고객 문의 대응 챗봇
- 학교 정보 안내 챗봇
- 매칭 결과 설명 봇

**투자자 시연 포인트**
- AI 챗봇이 고객 대응에 어떻게 활용되는지
- 학교 코디네이터와의 상호작용
- 실시간 답변 제공

### 5.3 기술 스택 정리

#### 백엔드 (Patrick Park)
- Spring AI
- RAG (Retrieval-Augmented Generation)
- Gemini API 연동
- Vector DB (추후)

#### 프론트엔드 (JI UN HONG)
- React 18 + TypeScript
- Tailwind CSS
- Recharts (매칭 시각화)
- Axios (API 통신)

- **리스크**: 매칭 테스트 불가
- **완화**: 샘플 데이터 5~10개로 먼저 진행

#### 정형화된 데이터 항목 불일치
- **리스크**: 학교마다 다른 포맷
- **완화**: 필수 항목과 선택 항목 분리

### 11.3 일정 리스크

#### 투자자 미팅 일정
- **리스크**: 프로토타입 완성 전 미팅 요청 가능
- **완화**: 핵심 플로우부터 완성도 높게 구현
