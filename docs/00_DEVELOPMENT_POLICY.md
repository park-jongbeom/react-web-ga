# 📜 AI Tool & Development Policy

## 1. 개발 기본 지침
1. 모든 응답은 한국어로 수행한다.
2. 외부 기능/알고리즘 구현 시 주석으로 출처(URL)를 명시한다.
3. **중대한 변화** (신기능, DB 수정, UI 대규모 변경) 시:
   - 전용 브랜치를 생성한다.
   - `docs/History/` 폴더에 변경 내역을 문서화한다.
4. 모든 기능 구현 시 해당 기능을 검증하는 **테스트 코드**를 반드시 포함한다.

## 2. 커밋 메시지 규칙 (상세 기술)
나중에 봐도 작업 맥락을 알 수 있도록 아래 항목을 충실히 작성한다.
- **내가 요구한 내용**: 유저의 요청 사항 요약
- **네가 답한 내용**: AI가 제안한 해결책 및 설계 요약
- **수정한 내용 요약**: 실제 파일 및 로직 변경 사항

## 3. Problem 1-Pager 양식 (코딩 전 필수)
문제가 복잡하거나 불명확할 경우 아래 양식을 작성하여 합의 후 진행한다.
- **배경(Background)**: 변경이 필요한 맥락과 동기
- **문제(Problem)**: 해결하려는 이슈의 본질
- **목표(Goal)**: 성공의 기준 (성공한 상태 정의)
- **비목표(Non-goals)**: 이번 작업 범위 밖의 사항 (명확히 제외)
- **제약(Constraints)**: 반드시 준수해야 할 기술적/비즈니스적 제약

## 4. Problem 1-Pager (UI 컴포넌트 템플릿화)
- **배경(Background)**: 버튼/입력/카드 등 UI가 페이지별로 분산되어 재사용성과 일관성이 낮음
- **문제(Problem)**: 동일한 UI 패턴이 중복되어 유지보수 비용이 증가
- **목표(Goal)**: 공용 UI 컴포넌트로 표준화하고 기존 화면을 해당 컴포넌트로 치환
- **비목표(Non-goals)**: 디자인 리뉴얼, 비즈니스 로직 변경, 백엔드 연동 변경
- **제약(Constraints)**: 기존 UX 유지, Tailwind 클래스 유지, 접근성 속성 보존
